# Data & Ingestion Pipeline

The Ingestion Engine is the mechanism TRACE uses to populate and maintain the underlying knowledge base from external sources (Semantic Scholar).

## The Pipeline Lifecycle

### 1. Triggering the Sync
A sync is triggered via the `/api/sync` admin endpoint.
To prevent database corruption if multiple admins attempt to trigger a sync concurrently, `backend/ingestion/ingestor.py` utilizes `portalocker` to claim an exclusive OS-level file lock on `data/sync_status.lock`. 

### 2. Incremental Fetching
For each person in the `people.json` registry, the ingestor checks their specific `last_year` timestamp stored in `data/sync_status.json`. 
When querying the Semantic Scholar Graph API, it passes `since_year=last_year`, meaning it **only fetches new papers** rather than re-downloading a researcher's entire historical catalog.

Results are **paginated** (100 per page). A single `limit=1000` request silently truncated prolific authors — no error, no log, just a permanent recall ceiling for anyone with a long publication record.

`MAX_PAPERS_PER_PERSON` (default 200, newest first) bounds how much is pulled per person, which bounds index size, embedding time and peak sync memory. Set it to `0` for unlimited, which is only sensible with an API key and plenty of RAM.

**Rate limiting**: the unauthenticated quota is 100 requests per 5 minutes, so a burst can only be cleared by waiting out a window measured in minutes. The client backs off 5s → 15s → 45s → 90s and honours `Retry-After` when the server sends it. Set `SEMANTIC_SCHOLAR_API_KEY` to raise the quota to 10 req/s.

#### Co-author metadata is merged, not overwritten
An incremental sync only refetches papers newer than each person's `last_year`, so a paper co-authored by A and B may come back under A alone. Upserting that record directly would overwrite `institute_authors` with just "A" and drop B from the paper **permanently** — degrading `people_to_consult` a little more on every sync.

Before upserting, the ingestor reads the currently stored `institute_authors` for each paper and merges the existing names back in. Only people still in the registry are re-added, so someone removed via `DELETE /api/people/{id}` is not resurrected by the next sync.

### 3. Metadata Parsing and Validation
As papers are retrieved, their metadata is aggregated (e.g. coalescing multiple authors from the same institute onto the same paper record).
Crucially, before a paper is embedded, its metadata is rigorously validated against the `DocumentMetadata` Pydantic model (`backend/rag/schemas.py`). 
This uses `extra="forbid"` to ensure no unexpected, corrupted, or unsupported fields are ever pushed into the Vector database.

### 4. Vector Embedding & Upsertion
Validated papers are passed to LangChain's Chroma integration. Embeddings are generated by the **Ollama** `all-minilm` model (`EMBEDDING_MODEL_NAME`), and records are upserted by `paper_id` so a re-sync is idempotent rather than duplicating.

Documents are stored as `Title: … \n\n Abstract: …`. Author names are *not* part of the embedded text — they are metadata, and are made searchable through the BM25 index instead (see the developer guide), which avoids a full re-embed whenever the roster changes.

### 5. Status Management
- **Success**: The `last_year` value is updated in the status file, and the total indexed paper count is refreshed.
- **Degraded**: If the Semantic Scholar API rate limits or errors out for more than 30% of the authors in the registry, the sync status is explicitly marked as `degraded`.
- **Evaluation**: Post-sync, `pipeline.post_sync_rebuild()` invalidates the semantic cache, rebuilds the BM25 index, and runs `self_retrieval` over a random sample of up to 200 abstracts. If `Hit Rate@5` falls below 90% the sync is marked `degraded`, warning of an embedding anomaly. The rate is returned to the caller so the check runs **once** per sync — it previously ran twice, in the rebuild and again in the API layer.

Note what this check is and is not: it verifies index *integrity* (right model, uncorrupted vectors, sane HNSW settings). It is close to trivially satisfiable — a document's own abstract should always retrieve it — so a passing score is not evidence of search quality. For that, see [llmops-evaluation.md](llmops-evaluation.md).

## Orphan Cleanup
Paper deletion is **not** handled during routine syncing. 
If an author is removed via `DELETE /api/people/{id}`, a synchronous cleanup is performed:
1. TRACE finds all papers where the deleted person was an author.
2. If they were the *only* institute author on the paper, the paper is deleted.
3. If they co-authored with other active institute members, the paper is kept, but their name is stripped from the `institute_authors` metadata.

Known limitation: `institute_roles` and `departments` are aggregated strings and are not rebuilt during removal, so a removed person's department may linger on co-authored papers until the next full re-sync.
